# AX診断 スコアリング仕様書 (draft v0.1)

> ステータス: **仕様書ドラフト（承認待ち）** — 本仕様の承認後に実装(F2スコアエンジン)に着手する。
> 対象: B1 ax-audit-platform / 作成: 2026-07-09

## 0. 目的

業務棚卸の結果を **再現性のある数値**で評価し、AI導入の優先順位を機械的に導く。
LLM は「業務記述の解釈・分類」のみに使い、**スコア算出はルールベース**とする（診断のブレ防止）。

## 1. 評価単位

- 評価対象 = 「業務」1件（例: 広告レポート作成、SEO記事制作）。
- 各業務を **4軸** で 0–100 にスコア化し、重み付き合成で「優先度スコア」を出す。

## 2. 4軸の定義と算出式

各軸はサブ指標（0–100）の重み付き平均。サブ指標の初期重みは `weights.yaml` で管理する。

### A1. 自動化適性 automation_fitness（高いほど良い）
| サブ指標 | 取得元 | 0–100 の目安 |
|---------|-------|-------------|
| 定型度 routineness | アンケート自己評価(1–5) | (値-1)/4×100 |
| データ入手性 data_availability | アンケート＋文書解析 | 電子化済=100 / 一部=50 / 紙=0 |
| 判断の単純さ decision_simplicity | LLM分類(定型/準定型/非定型) | 定型=100/準=50/非=0 |

初期重み: 定型度0.4 / データ入手性0.35 / 判断単純さ0.25

### A2. インパクト impact（高いほど良い）
`impact_raw = 月間頻度 × 1回あたり時間(h) × 従事人数` を全業務内で min-max 正規化して0–100。
- 対数圧縮を挟む: `score = normalize(log(1 + impact_raw))`（外れ値の支配を防ぐ）

### A3. 実装容易性 feasibility（高いほど良い）
| サブ指標 | 0–100 の目安 |
|---------|-------------|
| 既存ツールAPI有無 | あり=100 / 一部=50 / なし=0 |
| データ形式の構造化度 | 構造化=100 / 半構造=50 / 非構造=0 |
| 連携先システム数(少ないほど易) | max(0, 100 - 20×(接続数-1)) |

初期重み: API0.4 / データ形式0.35 / 連携数0.25

### A4. リスク risk（**高いほど悪い** — 優先度では反転利用）
| サブ指標 | 0–100 の目安(高=危険) |
|---------|---------------------|
| 誤り許容度の低さ | 低許容=100 / 中=50 / 高許容=0 |
| コンプラ機微度 | 個人情報/契約=100 / 社内=50 / 公開=0 |

初期重み: 誤り許容度0.5 / コンプラ0.5

## 3. 優先度スコアと4象限マップ

```
priority = w_impact*A2 + w_fit*A1 + w_feas*A3 - w_risk*A4
初期重み(weights.yaml): w_impact=0.35, w_fit=0.25, w_feas=0.25, w_risk=0.15
```
- 4象限マップ = 横軸 **実装容易性(A3)** × 縦軸 **インパクト(A2)**。
  - 右上（高影響×高容易）= 「今すぐPoC」象限。
- 上位10業務をランキング出力。

## 4. 重み変更の反映（受け入れ基準）

`weights.yaml` の値を変更するとスコア・順位が変わることをテストで担保する
（同一入力でも重み変更で順位が入れ替わるケースを1件以上用意）。

## 5. デジタルマーケ5業務プリセット（差別化）

広告運用 / レポート作成 / 入稿 / SEO記事制作 / SNS運用 は、質問項目とベンチマーク値
(`benchmarks.yaml`)をプリセット。汎用テンプレートも別途用意。

## 6. 出力の注記（必須）

ROI・削減率は「想定値であり実測ではない」注記を全数値に自動付与する
（`reduction_table.yaml` の想定値を使用）。
